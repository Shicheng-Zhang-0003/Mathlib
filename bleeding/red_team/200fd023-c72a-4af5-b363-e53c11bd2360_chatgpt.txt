# The biggest improvement: you introduced profiles

This is probably the strongest architectural decision.

You now have:

```
SCIENTIFIC
GRAPHICS
EMBEDDED
```

That is exactly how a serious numerical library should think.

Because there is no single "best math library."

Different users want:

### Scientific computing

Priority:

```
accuracy > speed
```

### Graphics/game engine

Priority:

```
throughput > exactness
```

### Embedded

Priority:

```
determinism > everything
```

Trying to satisfy all three with one implementation creates a mess.

Profiles solve that.

This is honestly one of the most "professional" additions in v10p5.

---

# The second major improvement: testing philosophy

This is where v10p5 feels like a real release candidate.

You added:

```
fuzz_god_mode.c
fuzz_boundary_gauntlet.c
```

This is exactly the direction I wanted.

Especially because numerical libraries fail in weird places.

Not:

```c
sin(1.0)
```

Nobody cares.

They fail in:

```c
sin(1e300)
```

or:

```c
sin(NAN)
```

or:

```c
matrix_inverse(almost_singular_matrix)
```

Your test structure shows you understand that.

One of the biggest barriers for a libm replacement is:

If you can say:

# Problem 2: CMake still has the v10 issue

This survived:


I would block a stable release on this.

Because:

```

```

```

```

Crash.

```cmake

```

# Problem 3: The header aggregation is risky

This:

is convenient.

But eventually:


For v11 I would want:

```

 ├── fft.c
 ├── linalg.c
 ├── tensor.c
```

Even if everything remains header-based.

Why?

Because when something breaks, you want isolation.

# The theme of v11A1

## "Engineering hardening + architecture consolidation"

The release question changes:

v10:

> "Can we build this?"

v10p5:

> "Can we organize this?"

v11A1:

> "Can another engineer depend on this?"

---

# 1. Public API audit (highest priority)

Before anything else, I would freeze the shape of the library.

Create a list:

```
Public API:
    trig/
    fft/
    linalg/
    tensor/
    stats/
    optimization/
```

For every public function:

Ask:

### Does the name communicate intent?

Example:

```c
ml_fast_sin()
```

vs

```c
ml_sin_approx()
```

The second one tells users something important.

---

### Does the function have defined behavior?

Example:

```c
ml_matrix_inverse(A)
```

What happens if:

* A is singular?
* A is NaN?
* A has dimension zero?
* memory is insufficient?

A professional library needs answers.

---

# 2. Establish the "mathlib contract"

v11A1 should introduce a document:

```
DESIGN_CONTRACT.md
```

Something like:

## Floating point policy

* IEEE-754 supported
* NaN propagation rules
* infinity behavior
* precision targets

---

## Memory policy

Example:

> Core APIs never allocate internally.

or:

> Allocation is permitted only in convenience APIs.

---

## Threading policy

Important.

Does this library guarantee:

```c
ml_fft_execute()
```

is thread-safe?

Can two threads call it simultaneously?

---

## Determinism policy

Very important for numerical software.

Does:

```c
ml_matrix_multiply(A,B)
```

produce identical output on:

* AVX2
* AVX512
* ARM NEON?

---

# 4. Numerical validation framework

This is the biggest "libm" transition.

You need a comparison harness.

Example:

```
validation/
    compare_libm.c
    ulp_error.c
```

Then:

```
mathlib sin
       |
       |
       v

glibc sin()
       |
       |
       v

calculate difference
```

Output:

```
sin:
max error:
0.72 ULP

average:
0.11 ULP

worst input:
x=938475928.123
```

Now your claims become evidence.

---

# 5. Error handling consistency

I would pick one pattern.

Currently you likely have a mix:

```c
double function(double x)
```

and:

```c
int function(..., result)
```

and:

```c
NULL return
```

v11A1 should decide.

My preference:

Core math:

```c
double ml_sin(double x);
```

because libm works this way.

Complex operations:

```c
ml_status ml_matrix_inverse(
    const ml_matrix *A,
    ml_matrix *result
);
```

because failure is possible.

# 8. Performance regression tracking

You already have benchmarks.

The important number is not:

The important number is:

"Did we accidentally become slower?"
